## Task A — BDS 17/19 Continued: Conceptual Deep-Dives

### 1. Why Use Sequencer for Real-Time Gameplay Events?

Sequencer is not limited to cinematic cutscenes. Through the **Gameplay Sequencer** feature, it can drive in-world events that occur while the player is actively playing — such as a gate collapsing when a button is pressed, or a spaceship crashing mid-level.

**Why Sequencer over Blueprint for these cases?**
- The animation path is predetermined and linear — no dynamic branching needed.
- Complex physics-driven animations (collapsing structures, falling vehicles) are computationally expensive to simulate in Blueprint using Timeline components or raw code.
- Sequencer handles these through the engine's internal keyframe evaluation system, which is significantly more GPU/CPU efficient for pre-authored motion.

**Takeaway for TA practice:**
> Blueprint → dynamic logic, player-driven conditions, randomized behavior.
> Sequencer → linear, complex, performance-stable animations — whether in cutscene or gameplay context.

### 2. Why Must Folder Structure Be Separated From Day One?

In a professional studio environment, multiple disciplines — Animators, Programmers, UI Designers, Level Designers — access the same project simultaneously via version control (Git, Perforce).

**What goes wrong without structure?**
- Placing `LevelSequence` files alongside `SkeletalMesh` assets in the same folder creates overlap zones.
- If two team members edit files in the same directory without clear ownership, Git **merge conflicts** occur.
- In binary asset pipelines, conflicts can corrupt files — hours or days of work lost.

**The TA's responsibility here:**
- Define and enforce a clean folder map before the team scales.
- Isolating cinematic assets into a dedicated `/Cinematics/` folder allows Animators to lock and work freely without disrupting Programmers working in `/Characters/`.
- This is one of the earliest and highest-leverage contributions a TA makes to a project's health.

### 3. Understanding the Root Cause of Double Transform Issues

During the tutorial, copy-pasting animation tracks between Stormtrooper actors worked cleanly — likely because both actors shared the same world-space origin or the tracks were copied from a neutral state.

**When does this break?**
When copying animation tracks from an actor already manually offset in the Level (e.g., `X=100`) to a second actor at a different position (e.g., `X=500`), Unreal may misinterpret the coordinate reference.

Unreal reads two distinct position types:
- **World Space** — absolute position on the map.
- **Local Space** — position relative to the actor's own origin or parent.

Raw track copying can cause the engine to additively stack both coordinates, resulting in a **double transform** (e.g., `X=600` instead of the intended `X=500`), or snap the target actor to the source position entirely.

**TA action:**
- Educate animators on this behavior proactively.
- Recommend using **Attach to Actor** or enabling **Relative Tracking** in Sequencer so animation data is portable across actors regardless of their world-space position.

### 4. Linear vs. Cubic Interpolation Curves — Choosing the Right Tool

| Curve Type | Behavior | Analogy |
|---|---|---|
| **Cubic (Auto/Smooth)** | Gradual acceleration at start, gradual deceleration at end (ease-in / ease-out) | A car pulling away from a stoplight, then braking smoothly at the destination |
| **Linear** | Constant velocity from first to last frame — no easing | A factory machine: instant start, instant stop |

**When to use each:**

- **Cubic** → Living creatures, camera movement, character limbs. Biological motion inherently contains acceleration and deceleration. Removing ease-in/out makes animated characters feel robotic and unconvincing.
- **Linear** → Mechanical objects with no inertia: automatic doors, elevators, helicopter rotors, conveyor belts. An elevator with a Cubic curve would feel wrong — it would appear to slow down mid-floor before stopping. Additionally, linear interpolation is computationally cheaper than cubic spline evaluation, which matters at scale.
