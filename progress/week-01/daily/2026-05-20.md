## Task A — BDS Part 18: Cinematic Pipeline Decisions

### 1. Camera Composition at the Start of the Production Phase

GPU memory (VRAM) is finite. Loading polygon data and textures outside the camera's field of view burns computational budget with zero contribution to the final frame.

Locking the camera early is a **Frustum Optimization** decision. By defining the rendering boundary upfront, aggressive culling can be applied to all out-of-frame assets.

| Impact Area | Result |
| --- | --- |
| Draw Calls | Reduced — only in-frustum geometry submitted |
| VRAM Allocation | Conserved — out-of-frame textures excluded |
| Polygon Density | Concentrated on pixels that reach the final output |

---

### 2. Console Command — `r.RayTracing.Culling 0`

The engine automatically culls ray tracing shadow calculations for objects beyond a defined radius — a real-time performance safeguard that causes visible shadow popping in offline renders.

In an **offline cinematic pipeline**, editor interactivity is deprioritized in favor of maximum visual fidelity. This command forces the GPU to execute full bouncing light calculations at extreme distances.

| Mode | Priority | Trade-off |
| --- | --- | --- |
| Default (Culling ON) | Interactive frame rate | Shadow popping at distance |
| `r.RayTracing.Culling 0` | Visual fidelity maximum | Interactive performance sacrificed |

---

### 3. Look-At Tracking with Relative Offset on the Z-Axis

The engine's native Look-At Tracking reads only the pivot point of a target object. On most character assets, the pivot sits at ground level (0, 0, 0) for ground snapping — causing auto-tracking to aim at the character's feet by default.

The **Relative Offset on the Z-axis** functions as a data interceptor — shifting the tracking matrix upward without breaking the automated dependency chain.

| Method | Behavior | Risk |
| --- | --- | --- |
| Manual camera rotation | Full control | Framerate jitter from imperfect curve interpolation |
| Look-At Tracking (raw) | Automated, smooth | Aims at pivot — usually at ground level |
| Look-At + Z Offset | Automated, smooth, correct focal point | None |

---

### 4. Play Rate 0.5 and Linear Interpolation

Raw animation assets are produced at 1:1 real-time scale. Massive objects — aircraft, troop formations — lose their sense of visual weight at full speed when the camera moves dynamically across large environments.

Reducing play rate to `0.5` doubles the data samples per movement (**Frame Allocation Expansion**). Converting curves to **Linear** removes Ease-In/Ease-Out, guaranteeing constant velocity between keyframes and eliminating velocity jumps across camera cuts.

| Curve Type | Behavior | Use Case |
| --- | --- | --- |
| **Cubic (Auto/Smooth)** | Ease-in at start, ease-out at end | Organic motion: characters, camera, biological objects |
| **Linear** | Constant velocity — no easing | Mechanical objects: elevators, rotors, conveyor belts |

---

### 5. Character Component Isolation — Disabling `Receive Decal`

The Decal system projects material data using a bounding box volume. Any dynamic object intersecting that volume is forced to receive decal material calculations on top of its own base material — causing unintended color bleed onto character surfaces.

Disabling `Receive Decal` on the character Skeletal Mesh severs the rendering pass instruction chain at the object level, excluding the character's material ID from the decal projection pass entirely.

| Approach | Visual Result | Cost |
| --- | --- | --- |
| Reposition decal bounding box | Floor integrity compromised | Free |
| Disable `Receive Decal` on Skeletal Mesh | Clean isolation, floor intact | Zero additional computation |
