## Task A — Computational Efficiency: Performance Budget Decision

### Lumen vs. Path Tracing — Rendering Cost Analysis

GPU time is a non-renewable resource per frame. Every rendering decision must be weighed against the project's final delivery target, not against theoretical maximum quality.

Two rendering pipelines were evaluated in the final BDS session:

| Pipeline         | Render Time             | Visual Quality                    | Appropriate Use Case                         |
| ---------------- | ----------------------- | --------------------------------- | -------------------------------------------- |
| **Lumen**        | ~4 min / 100+ frames    | Real-time GI — minor noise        | Daily iteration, social media delivery       |
| **Path Tracing** | Hours / GPU (RTX 4090)  | Noise-free, physically accurate   | Final cinematic baking, print-quality stills |

**TA Decision Logic:**

If the perceptual delta between Lumen and Path Tracing is negligible to the end consumer's eye, shipping a Path Tracing pipeline into the team's main repository is an unjustifiable hardware budget expenditure. The correct TA decision is to default to Lumen for real-time and iterative work, reserving Path Tracing exclusively for final delivery frames where the quality difference is contractually mandated.

> Submitting a render pipeline that is not reproducible within the team's hardware budget is a pipeline integrity failure — not a quality improvement.

## Task B — Pipeline Integrity: Camera Lock & System Glitch Resolution

### Camera Actor Transform Lock in Sequencer

In multi-member productions, any unlocked actor in a shared level is a liability. An animator or level designer opening the scene the following morning can inadvertently shift a camera that has already been director-approved — destroying framing without any audit trail.

**Preventive measure applied:** `Transform → Lock Actor Movement` on the camera actor inside Sequencer before committing the scene.

| Scenario                        | Without Lock                               | With Lock                                          |
| ------------------------------- | ------------------------------------------ | -------------------------------------------------- |
| Team member opens level         | Camera position can be accidentally moved  | Transform is read-only — framing preserved         |
| Director-approved shot is saved | No protection against accidental edits     | Coordinate values frozen until explicitly unlocked |

**System Glitch Resolution:**

During the lock procedure, Unreal Engine's right-click context menu failed to surface the lock option — a known transient cache issue. TA decision: do **not** force-edit raw asset files to compensate for a UI anomaly. Instead, perform a safe project restart to flush memory state, then re-execute the operation on a clean session. This preserves data structure integrity before committing to the repository.

> Forcing operations on a glitched editor state is a pipeline risk. A clean restart costs two minutes — recovering from corrupt data costs days.

## Task C — System Automation: Console Variable Presets & Output Naming

### Console Variable Presets (CVar Automation)

Manual console command entry (e.g., `r.RayTracing.Nanite.Mode 1`, `r.ScreenPercentage`) introduces two failure modes: human typo errors and inconsistent configuration states across team workstations.

**Solution implemented:** All required graphics variables saved into a single named preset file via `Console Variables → Save Preset As`.

| Approach                | Time Cost    | Error Risk             | Team Scalability           |
| ----------------------- | ------------ | ---------------------- | -------------------------- |
| Manual console entry    | Per-session  | High (typo, omission)  | Non-scalable               |
| CVar Preset (one-click) | Once         | Near-zero              | Full team reproducibility  |

Any team member can now load the correct rendering state in a single click, without requiring knowledge of the underlying console variable syntax. The system configures itself.

### String Expression Output Naming

Static filename prefixes break when sequence names change during production. If the sequence is renamed mid-project, all previously rendered output becomes misidentified and pipeline tools (compositing, video editing software) lose their frame order reference.

**Solution implemented:** Output filename format set to a code expression: `SD.{FrameNumber}` (example pattern).

| Naming Method               | Behavior on Sequence Rename                     | Frame Order Integrity |
| --------------------------- | ----------------------------------------------- | --------------------- |
| Static prefix               | Breaks — stale naming                           | Lost                  |
| `{FrameNumber}` expression  | Sequence-agnostic — auto-increments (001, 002…) | Always intact         |

The naming system is now decoupled from human decisions. Downstream post-production tools receive correctly ordered, consistently named files regardless of upstream production changes.
